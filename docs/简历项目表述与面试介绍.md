# ZAgent 项目简历表述与面试介绍

本文档基于当前仓库实际实现整理，目标是：

- 帮你把项目写得**准确、能自圆其说、不过度夸大**
- 帮你把每个项目点压成**简历可写版**和**面试可口述版**
- 帮你知道每一条大概该看哪些源码

## 一、先给结论：这项目最适合怎么定位

不建议表述成：

- 完整自治智能体平台
- 通用多智能体框架
- 从 0 自研大模型基础设施

更稳的定位是：

- 基于 Spring AI 的 AI Agent 后端编排系统
- 面向多角色、多工具场景的配置驱动型 Agent 系统
- 支持动态装配、RAG 检索增强、MCP 工具接入和多阶段执行链

一句话项目描述建议：

> 基于 Spring AI 构建的 AI Agent 后端系统，将 Model、Prompt、Advisor、MCP 工具等能力组件配置化，支持 RAG 检索增强、MCP 工具调用以及 Fixed / Flow / Auto 多种执行策略。

## 二、建议的简历版本

下面是一版更贴合当前代码实现、同时又适合面试的写法。

### 项目描述（建议版）

基于 Spring AI 构建 AI Agent 后端系统，将 Model、Prompt、Advisor、MCP 工具配置化并持久化管理，支持 RAG 检索增强、MCP 工具调用以及多阶段 Agent 执行链。

- 基于配置驱动装配机制动态构建 `ChatClient`，按 `clientId` 运行时组装 Model、Prompt、Advisor、MCP 工具，并通过本地缓存复用已构建实例

### 如果你想写得再保守一点

- 基于配置驱动动态装配 `ChatClient`，支持模型、提示词、增强器与 MCP 工具的运行时组合
- 基于 `PgVector + Tika` 接入 RAG 检索链路，实现知识入库、相似检索与上下文增强
- 接入 MCP 工具协议，支持 `stdio` / `sse` 两种工具调用方式
- 实现多种 Agent 执行模式，支持固定流程执行、动态规划执行与多轮协同执行

## 三、每一条项目点该怎么写、怎么讲

## 1. 动态装配 ChatClient

### 简历怎么写

建议写法：

> 基于配置驱动装配机制动态构建 `ChatClient`，按 `clientId` 运行时组装 Model、Prompt、Advisor、MCP 工具，并通过本地缓存复用已构建实例

### 为什么这样写

因为当前代码里，确实存在：

- 运行时按 `clientId` 查询绑定关系
- 组装 `OpenAiApi`、`OpenAiChatModel`、`Advisor`、`MCP Tool`
- 产出 `ChatClient`
- 用 `ConcurrentHashMap` 做单机缓存

但不建议写成：

- 自研 AI 装配框架
- 自研多租户 Agent 平台
- 支持配置热更新一致性切换

因为这些都超过当前实现边界。

### 面试怎么介绍

30 秒版：

> 我做的是基于配置驱动的 `ChatClient` 动态装配。因为项目里有多角色、多模型、多工具组合，如果把 Model、Prompt、Advisor、MCP 都写死在代码里，扩展成本会很高。所以我把这些组件关系持久化到数据库，请求进来后按 `clientId` 查询配置，在运行时组装成对应的 `ChatClient`。为了减少重复构建，又在装配层做了本地缓存复用。

1 分钟版：

> 我这个项目里，`ChatClient` 不是在代码里写死的，而是按配置动态构建的。原因是系统里有不同角色和执行节点，比如规划、执行、监督，不同节点可能依赖不同的模型、Prompt、Advisor 和工具。如果每种组合都单独写一套代码，后面会很难维护。所以我把 Model、Prompt、Advisor、MCP 的绑定关系放到数据库里，请求进来后根据 `clientId` 查询并组装成 `ChatClient`。这样做的价值主要有三个：第一，减少硬编码；第二，支持多角色能力复用；第三，后续调整模型或工具绑定时，不需要改主业务流程。当前运行时还做了基于 `ConcurrentHashMap` 的本地缓存来减少重复装配开销。

### 面试高频追问怎么答

#### 为什么放数据库，不放 YAML / Redis / Nacos？

> 因为这些在项目里更像业务运行配置，不是静态系统配置。它们之间有模型、Prompt、Advisor、工具绑定的关联关系，而且需要后台动态调整，数据库更适合做关系建模和持久化管理。YAML 更适合静态配置，Redis 更适合缓存，Nacos 更适合分发型配置，不太适合直接承载复杂关系实体。

#### 缓存是怎么做的？

> 当前是单机内存缓存，按 `clientId` 缓存已组装好的 `ChatClient`，第一次请求构建，后续直接复用。实现上用的是 `ConcurrentHashMap`。

#### 配置变更后缓存怎么失效？

> 当前实现里已经有手动失效入口，但还没有完全做成统一的配置变更联动机制。更完整的做法应该是：配置更新成功后，找到受影响的 `clientId`，主动删除对应缓存；如果是多节点，再通过消息或广播机制同步失效。

#### 如果配置非法怎么办？

> 配置不能默认可信，所以应该分成两层：保存配置时做前置校验，运行时装配时再做兜底校验。如果新配置不合法，不应该直接把旧缓存删掉强切，而应该优先保留最后一个可用版本。

## 2. RAG 检索增强链路

### 简历怎么写

建议写法：

> 实现基于 `PgVector` 的 RAG 检索增强链路，文档经 `Tika` 解析、切块与向量化后入库，结合自定义 `RagContextAdvisor` 支持按标签过滤的 TopK 相似检索与上下文注入

### 为什么这样写

当前仓库里确实实现了：

- 文档上传
- `Tika` 解析
- 分块
- 元数据打标
- 向量入库到 `PgVector`
- 通过 `RagContextAdvisor` 做检索与上下文注入

但不建议写成：

- 从 0 自研 RAG 引擎
- 自研向量检索系统
- 自研召回排序框架

### 面试怎么介绍

30 秒版：

> 我在项目里接入了一条完整的 RAG 链路。文档上传后先用 `Tika` 做解析，再做切块和向量化，最后存入 `PgVector`。查询时不是直接把问题发给模型，而是先做相似检索，再通过自定义 `RagContextAdvisor` 把召回结果注入模型上下文，增强回答的事实依据。

1 分钟版：

> 我这个项目里的 RAG 不是从 0 自研底层向量检索，而是基于 Spring AI 和 `PgVector` 做工程化落地。整体流程是：文档上传后先用 `Tika` 解析文本，再做切块，把每个 chunk 连同 `knowledge`、`source` 这类元数据一起入库。查询时，根据用户问题做相似检索，如果指定了知识标签，还会带上过滤表达式，只在对应标签范围内做 TopK 相似召回。最后通过自定义的 `RagContextAdvisor` 把检索结果注入到模型上下文中，让模型基于外部知识回答，而不是只依赖参数记忆。

### 面试高频追问怎么答

#### 你的 RAG 是从 0 做的吗？

> 不是从 0 造底层向量引擎，而是基于现有框架做工程化接入。我的重点是把文档上传、解析、切块、向量入库、检索召回和上下文注入这条链路打通，并接进 Agent 回答流程。

#### 什么是按标签过滤的 TopK 相似检索？

> 就是先通过元数据标签缩小候选集合，再在这个集合里做向量相似度排序，最后取前 K 个结果。不是在全量库混着查。这样能减少不同知识域之间的相互干扰。

#### RAG 和模糊搜索有什么区别？

> 模糊搜索更偏关键词匹配，RAG 更偏语义检索加生成。RAG 不一定一定更高效，强关键词和结构化检索场景下传统搜索可能更直接；但在自然语言问答和非结构化文档场景里，RAG 更有优势。

#### 怎么提高召回率？

> 重点一般在切块策略、Query 改写、混合检索、TopK 调参、相似度阈值、以及必要时加 rerank，而不只是换一个 embedding 模型。

## 3. MCP 工具接入层

### 简历怎么写

建议写法：

> 实现 MCP 工具接入层，支持 `stdio` / `sse` 两种传输模式，并将外部 MCP 能力统一转化为可供 Spring AI 调用的 Tool 能力

### 为什么这样写

当前实现里，确实做了：

- MCP 配置同步
- MCP 绑定解析
- `stdio` / `sse` 两种传输解析和接入
- 将 MCP Tool 转成 `ToolCallback` 参与 `ChatClient` 调用链

但不建议写成：

- 自研 MCP 协议
- 自建 MCP 市场
- 实现复杂分布式工具编排平台

### 面试怎么介绍

30 秒版：

> 我在项目里接入了 MCP 工具协议，主要做的是把外部 MCP 服务能力纳入 Agent 调用链。当前支持 `stdio` 和 `sse` 两种传输方式，后端会根据配置解析工具绑定关系，创建对应的 MCP Client，并把 MCP 暴露出来的工具统一转成 Spring AI 可调用的 Tool 能力。

1 分钟版：

> 我做的 MCP 部分，核心不是自己造协议，而是把 MCP 作为统一工具接入层纳入到 Agent 系统里。当前支持两种传输方式：本地进程型的 `stdio`，以及远程服务型的 `sse`。后端会先同步和解析 MCP 配置，再根据 client 绑定关系决定当前 `ChatClient` 可以使用哪些工具。工具层面则通过 Spring AI 的 MCP Tool Callback 机制，把外部 MCP 服务统一转成模型可调用的 Tool 能力。这样做的好处是工具接入和业务流程解耦，后续新增工具不需要改主执行链。

### 面试高频追问怎么答

#### 为什么要接 MCP？

> 因为模型本身只会生成文本，MCP 提供了一套统一工具协议，便于把文件、天气、地图等外部能力接进来，让 Agent 不只是会说，还能调用外部能力完成任务。

#### `stdio` 和 `sse` 的区别？

> `stdio` 更适合本地进程型工具，调用时通过标准输入输出和子进程通信；`sse` 更适合远程服务型工具，通过 HTTP 长连接接收事件流。前者更像本地插件，后者更像远端工具服务。

#### 你项目里怎么知道某个工具有没有真的被调用？

> 当前已经加了调用日志，能看到工具调用开始、成功与结果预览；另外从 `Flow` / `Auto` 的执行日志和返回内容里也能判断工具是否实际参与了回答链路。

#### 这块最大的坑是什么？

> 比较典型的是配置错误、SSE endpoint 不匹配、stdio 进程无法启动、工具参数不规范，以及工具返回结果格式不稳定。这些问题会直接影响 Agent 运行时的工具可用性。

## 4. Flow / Auto 执行链

### 简历怎么写

建议写法：

> 实现 `Fixed` / `Flow` / `Auto` 三种 Agent 执行策略，其中 `Flow` 采用固定执行阶段下的动态步骤规划，`Auto` 采用基于角色分工与反馈闭环的多轮协同执行

### 为什么这样写

当前代码里：

- 确实存在 `FixedExecuteStrategy`
- 确实存在 `FlowExecuteStrategy`
- 确实存在 `AutoExecuteStrategy`

其中：

- `Fixed` 是固定顺序执行
- `Flow` 是固定阶段（工具分析、规划、执行）+ 动态规划步骤
- `Auto` 是固定角色结构（分析、执行、监督、总结）+ 多轮反馈闭环

所以不建议写成：

- 预定义流程编排平台
- 通用工作流引擎
- 全自治多智能体系统

### 面试怎么介绍

30 秒版：

> 我在项目里实现了三种执行策略：`Fixed`、`Flow`、`Auto`。`Fixed` 适合固定顺序场景；`Flow` 适合需要先分析工具、再规划步骤、再分步执行的任务；`Auto` 更偏多轮协作，按分析、执行、监督、总结几个角色推进任务。这样做是为了让系统在不同复杂度任务下有不同执行方式，而不是所有请求都走同一条链路。

1 分钟版：

> 我把 Agent 执行链拆成了三种策略。`Fixed` 比较简单，就是按数据库配置好的顺序串行执行，适合流程固定的场景。`Flow` 不是把步骤完全写死，而是固定“工具分析、规划、执行”这几个执行阶段，再由模型根据当前任务动态生成执行步骤，后端再解析这些步骤并按步骤推进。`Auto` 则更偏多轮协同，结构上固定了分析、执行、监督、总结这几个角色，通过监督结果决定是否继续迭代。也就是说，这三种策略不是三个完全独立的系统，而是三种不同复杂度和控制方式的执行链设计。

### 面试高频追问怎么答

#### `Flow` 是不是预定义流程编排？

> 更准确地说，`Flow` 是固定执行阶段下的动态步骤规划，不是通用工作流引擎。阶段是预定义的，但具体步骤是模型根据任务生成、后端再解析执行的。

#### `Auto` 和 `Flow` 有什么区别？

> `Flow` 更像“先规划，再按步骤执行”；`Auto` 更像“角色分工 + 多轮反馈闭环”。前者重在步骤规划，后者重在监督和迭代。

#### 你们 `Flow` 的规划怎么解析？

> 现在做的是 JSON 优先解析，失败时回退到文本解析。这样比纯文本正则更稳，也便于后续做结构化校验。

#### 为什么要做多种策略，不统一一种？

> 因为不同任务复杂度不同。简单场景用 `Fixed` 更稳；需要分步规划的用 `Flow`；需要多轮协作和质量反馈的用 `Auto`。统一一条链路会要么过重，要么不够用。
