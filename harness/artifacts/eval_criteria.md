# Evaluation Criteria

本文件定义 `Evaluator` 的固定评分维度与硬阈值。
只要任一项低于阈值，则本轮 Sprint 必须判定为 `FAIL`。

## 1. 产品深度（Product Depth）
问题：本轮是否真正推进了产品体验，而不是只做了表面 UI 或占位逻辑？

评分参考：
- 1-2：大多是表层实现，缺少实际产品价值
- 3：有一定用户价值，但仍较浅
- 4：提供了明确且真实的产品推进
- 5：这一轮明显提升了产品能力与可用性

阈值：4

## 2. 功能完成度（Functionality）
问题：合同承诺的核心流程是否真实可用、端到端闭环是否成立？

评分参考：
- 1-2：核心流程不可用或严重损坏
- 3：部分可用，但关键问题仍存在
- 4：核心流程可用，剩余问题不阻塞主路径
- 5：功能可靠，关键路径运行稳定

阈值：4

## 3. 视觉与体验质量（Visual Design / UX Quality）
问题：界面是否清晰、交互是否连贯、体验是否达到可接受水平？

评分参考：
- 1-2：混乱、粗糙、难以使用
- 3：基本可用，但体验较粗糙
- 4：体验清晰、一致，达到可接受质量
- 5：体验成熟、自然、有明显打磨痕迹

阈值：3

## 4. 代码质量（Code Quality）
问题：实现是否聚焦、结构是否可维护、是否与合同范围一致？

评分参考：
- 1-2：结构混乱、脆弱、明显超范围或难以维护
- 3：基本可接受，但存在明显结构问题
- 4：结构清晰、聚焦、可维护
- 5：代码质量高，边界清晰，便于后续演进

阈值：4
