# Product Spec

## 文档状态
- 状态：已确认
- 最后更新时间：2026-03-26
- 当前阶段：实施中

## 一、产品愿景
- `zagent` 是一个面向演示、学习与简历展示的 Agent Studio，用来集中展示 Agent 编排、工具调用、知识增强与可观测性能力。
- 它不再以“资源配置后台”为主叙事，而是以“如何设计、运行、观察一个 Agent 系统”为主叙事，帮助面试官和使用者快速理解项目价值。

## 二、目标用户与角色
- 核心用户：准备 Agent / AI 应用开发岗位的开发者，即项目作者本人。
- 次级用户：面试官、技术评审、对 Agent 工程化感兴趣的同行。
- 核心需求：快速理解项目架构；看到可运行的 Agent 演示；理解不同策略范式；看到工具、Prompt、RAG 与运行状态如何协同。

## 三、核心使用场景
- 场景 1：在首页快速理解项目能力边界、策略范式与演示入口。
- 场景 2：在 Playground 中直接发起一次 Agent 运行，观察输入、策略、工具与输出。
- 场景 3：在 Settings 中查看项目底层配置能力，但不被配置细节淹没。

## 四、关键用户旅程
### 旅程 1：面试演示
- 用户起点：进入系统首页。
- 用户动作：阅读能力说明，进入 Playground，选择策略并触发运行。
- 系统反馈：展示系统定位、核心能力、快捷入口与演示界面。
- 成功结果：用户能在 3 分钟内理解项目亮点与技术深度。

### 旅程 2：项目讲解
- 用户起点：进入导航主界面。
- 用户动作：查看 Workflows、Tools、Knowledge、Observability、Settings 等入口。
- 系统反馈：按产品能力组织页面，而不是按后台资源组织页面。
- 成功结果：用户能用产品语言讲清系统设计，而不是陷入零散配置讲解。

## 五、主要功能模块
- `Overview`：展示产品定位、核心能力、关键指标、推荐演示路径。
- `Playground`：统一承载聊天测试、Agent 运行、策略体验等主演示流。
- `Workflows`：承载 Agent 策略与执行编排视角。
- `Tools`：承载工具调用与 MCP 工具能力入口。
- `Knowledge`：承载 RAG 与知识库能力入口。
- `Observability`：承载运行过程、步骤、工具调用、状态与后续指标展示。
- `Settings`：承载模型、Prompt、API、Advisor、MCP、Client、Agent 等配置页。

## 六、高层技术架构
- 前端形态：Vue 3 + Vite 单页应用，采用 Agent Studio 产品壳。
- 后端形态：Spring Boot 模块化单体，后续按领域边界重构。
- 数据存储：延续当前数据库与向量存储方案。
- 集成能力：OpenAI / Spring AI、MCP、RAG、可配置 Prompt 与模型装配。
- 部署形态：本地开发与演示优先，保持单体启动方式。

## 七、后端领域模块边界（规划）
目标：让后端结构与“Agent Studio”叙事一致，避免按技术层堆文件导致不可讲解。

建议领域模块（模块化单体，不等于 Maven 多模块）：
- `agent`：策略选择、执行编排、运行入口契约（Plan/Execute、ReAct、Reflection 等）。
- `tooling`：工具注册、工具调用回调、MCP 绑定与工具形态抽象。
- `knowledge`：RAG、向量检索、知识库标签与生命周期。
- `runtime`：运行态状态、会话/上下文、执行中间产物（后续承接 Observability 的数据源）。
- `observability`：trace/step/tool-call 记录与查询接口（先定义边界，后续实现）。
- `configcenter`：模型/Prompt/Advisor/API/Client 等装配与配置管理能力（产品上降级为“支撑能力”）。
- `web`：对外 API（Controller / DTO / VO）与前端契约，逐步从“散落控制器”收敛成领域入口。

## 七、AI 能力机会
- Agent 策略展示：用统一入口展示 `Plan and Execute`、`ReAct`、`Reflection` 等范式。
- Tool Use 展示：将工具调用能力从隐藏配置提升为用户可感知能力。
- 可观测性：让用户能看到运行过程，而不仅看到最终答案。

## 八、演示叙事（简历/面试用）
面向面试的最短演示路径（North Star Demo）：
- `Overview`：一句话定位 + 核心能力列表 + 推荐演示路径。
- `Playground`：选择策略（如 ReAct / Plan&Execute / Reflection）→ 输入任务 → 触发运行 → 查看输出与关键过程摘要。
- `Settings`：只在需要解释“为什么能做到”时，进入查看模型、Prompt、Tool、RAG 等支撑配置。

## 九、质量要求
### 功能性
- 首页、导航、Playground、Settings 入口必须可访问。
- 主叙事必须从“管理后台”切换为“Agent Studio”。

### 体验
- 视觉风格应现代、简洁、偏产品演示，不得继续呈现传统后台观感。
- 主入口、页面命名与说明必须有助于演示与面试讲解。

### 性能
- 首轮重构不引入重型前端依赖。
- 保持当前本地构建与启动成本可接受。

### 稳定性
- Sprint 1 不破坏已有关键配置页访问能力。
- 所有变更需通过前端构建验证。

## 十、范围边界（非目标）
- 不追求在 Sprint 1 内完成“全量功能重做”。优先把信息架构、叙事和主演示流立住。
- 不追求在 Sprint 1 内把后端彻底模块化，只做前端壳层与入口重组。
- 不以“配置项数量”作为产品价值指标，配置能力仅作为支撑系统能力的必要手段。

## 十一、风险与未决问题
- 风险 1：现有页面较多，若一次性全部重写会导致 Sprint 1 过大。
- 风险 2：当前中文文案与结构存在历史编码痕迹，需顺手修正。
- 未决问题：Observability 页面真实数据接入放到后续 Sprint，不在本轮强行完成。
